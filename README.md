# auth-impact-poc

## Purpose

SPA における認証方式の違いが、

- フロントエンド実装
- 仕様変更時の影響範囲
- 運用・保守コスト

にどのような差を生むかを検証する PoC。

## Background

保守・改修フェーズの Web サービスでは、
「認証まわりがブラックボックス化しており、変更のたびに調査コストが発生する」
状態がボトルネックになりやすい。

本 PoC では、**実装方式の違いが将来コストにどう影響するか**を
フロントエンド視点で整理・比較する。

## Scope

- SPA 構成における認証方式の違い（例：Cookie / Token）
- 認証状態と画面制御の責務分離
- 仕様変更時に影響を受ける範囲の整理

※ 技術選定そのものが目的ではなく、
「変更に強いか」「説明可能か」を評価軸とする。

## Output

- 実装パターンごとの構成整理
- 変更シナリオ別の影響範囲メモ
- 保守・改修フェーズでの判断材料
